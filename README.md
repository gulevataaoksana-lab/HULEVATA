Лабораторна робота №1

- Я використала теги "HTML5" для створення чіткої ієрархії документа: `header` для заголовка та панелі керування, `main` для основного контенту та `section` для логічного поділу форми й таблиць.

- Таблиці реалізовані за всіма правилами семантики: з використанням блоків `thead` (для заголовків), `tbody` (для динамічних даних) та тегів `th`, що забезпечує коректну структуру даних.

- Кожному полю введення (`input`, `select`, `textarea`) я призначила унікальний `id` та зв’язала його з відповідним елементом `label` через атрибут `for`.

- Згідно з варіантом №6, я створила форму для репорту вразливості, що містить **6 полів**: текстові інпути для ID та імені автора, назву вразливості, випадаючі списки для вибору рівня критичності й статусу, а також `textarea` для розширеного опису.

- Кнопки форми мають чітко визначені типи: `type="submit"` для ініціації валідації та додавання запису, та `type="button"` для допоміжних дій (очищення форми), що запобігає випадковому перезавантаженню сторінки.

- Для побудови інтерфейсу я використала "Flexbox". Головний контейнер `.app` має `display: flex`, що дозволило мені реалізувати двоколонну структуру ("форма зліва, списки справа").

- Для елементів керування в шапці (`topbar`) я застосувала `justify-content: space-between`, рознісши пошук та фільтри по краях, та використала `flex: 1` для однакового розподілу ширини між кнопками.
-* Я встановила глобальне правило `box-sizing: border-box`, щоб внутрішні відступи (`padding`) та межі (`border`) не змінювали розрахункову ширину елементів, забезпечуючи стабільність сітки.

 


Лабораторна робота №2

- Я перевела проєкт на "TypeScript", налаштувавши `tsconfig.json` та збірку через `ts-node-dev` для розробки. Це дозволило мені уникнути використання `any` та впровадити сувору типізацію для всіх DTO та бізнес-сутностей.

- Розділила код на логічні шари згідно з принципами "Clean Architecture": Routes/Controllers (прийом запитів), Services (бізнес-логіка), Repositories (in-memory доступ до даних) та DTOs (опис структур запитів та відповідей).

- Впровадила стандартизацію контрактів: усі відповіді для списків мають уніфікований формат `{ items: [...] }`, а помилки повертаються в єдиному форматі з кодом та деталями.

- Реалізувала повний набір CRUD-маршрутів для всіх сутностей варіанта №6: Користувачі (Users), Статуси (Statuses) та Звіти (Reports).

- Я дотрималась REST-стандартів: використання множини в URL (`/api/reports`) та відповідних HTTP-методів: GET (читання), POST (201 Created), PUT (200 OK) та DELETE (204 No Content).

- Для ідентифікації об'єктів реалізувала генерацію унікальних ID на бекенді, що гарантує цілісність даних у пам'яті.

- Розробила систему валідації вхідних даних (DTO): перевірка обов'язкових полів, відповідність типів (наприклад, формат дати ISO) та контроль допустимих значень для полів `severity` та `status`.

- Мною реалізована централізована обробка помилок через middleware. Будь-яка помилка (400 Bad Request, 404 Not Found або 500 Internal Server Error) перехоплюється та повертається клієнту у стабільному JSON-форматі.

- Додала логування кожного запиту: у консоль виводиться метод, шлях, код відповіді та час виконання запиту в мілісекундах.

- Реалізувала можливість фільтрації списку звітів через query-параметри (наприклад, `GET /api/reports?status=Новий`) та функціонал текстового пошуку за назвою вразливості або автором.




Лабораторна робота №3

Як запустити проєкт
1. Встановити залежності: "npm install"
2. Запустити сервер у режимі розробки: "npm run dev"
3. Сервер автоматично підніметься на "http://localhost:3000"

Робота з базою даних
- Файл бази даних створюється автоматично у папці `./data` під ім'ям `database.db`.
- Папка `./data` створюється при першому запуску, якщо її ще немає.

База даних містить три доменні таблиці та таблицю для міграцій.

1. Таблиця users (Користувачі):
   - `id`: TEXT PRIMARY KEY
   - `name`: TEXT NOT NULL UNIQUE

2. Таблиця statuses (Статуси):
   - `id`: INTEGER PRIMARY KEY AUTOINCREMENT
   - `name`: TEXT NOT NULL UNIQUE
   - `description`: TEXT

3. Таблиця reports (Звіти):
   - `id`: INTEGER PRIMARY KEY AUTOINCREMENT
   - `title`: TEXT NOT NULL
   - `severity`: TEXT NOT NULL CHECK (severity IN ("Низький", "Середній", "Високий", "Критичний"))
   - `status_id`: INTEGER NOT NULL
   - `description`: TEXT
   - `reporter_id`: TEXT
   - `createdAt`: DATETIME NOT NULL
   - FOREIGN KEY(`status_id`) REFERENCES `statuses`(`id`) ON DELETE RESTRICT
   - FOREIGN KEY(`reporter_id`) REFERENCES `users`(`id`) ON DELETE SET NULL

Приклади запитів (API Endpoints)
- Отримання всіх звітів з фільтром статусу: `GET /api/reports?status_id=2`
- Повний список із JOIN на авторів: `GET /api/reports`
- Створення звіту: `POST /api/reports`
  - Тіло: `{"title":"Помилка","severity":"Високий","status_id":1,"description":"Тест","reporter_id":"u1"}`
- Експорт у JSON: `GET /api/reports/export?status_id=2&limit=10&sort=createdAtAsc`

Демонстрація SQL Injection
У проєкті спеціально залишено ендпоїнт `/api/search-unsafe`, який використовує пряму конкатенацію рядка в SQL-запит.
Приклад ін'єкції: `http://localhost:3000/api/search-unsafe?title=1" OR "1"="1`
Запит перетвориться на `SELECT * FROM reports WHERE title = "1" OR "1"="1"`, і сервер поверне всі записи.

Файл SQLite зберігається у `./data/database.db`.




Лабораторна робота №4
1. Як запустити
Бекенд:
-Перейти в папку backend.
-Запустити сервер: npm run dev (за замовчуванням на порту 3000).

Фронтенд:

-Перейти в папку frontend.
-Запустити через Live Server (або інший локальний сервер) на порту 5500.

Впроваджено префікс /api/v1/ для всіх ендпоінтів.

Правила сумісності DTO:
- Незмінність: Назви існуючих полів (наприклад, title, severity) ніколи не змінюються, щоб не зламати клієнт.
- Розширюваність: Нові дані додаються лише як необов'язкові поля.
- Стабільність типів: Поле, що було числом (id), ніколи не стає рядком у межах однієї версії.

2. TypeScript та типізація
Весь фронтенд написаний на TypeScript (файл app.ts).

Використано інтерфейси для DTO (Report, User, Status), що забезпечує повну відповідність структурі даних з бекенда.

3. Реалізовано автоматичний таймаут запитів (10 секунд) за допомогою AbortController. Якщо сервер не відповідає, запит скасовується, а користувач бачить попередження.

Сценарії перевірки 
1. Успішна робота 
Дія: Запустити бекенд та відкрити сторінку.

Очікуваний результат: Статус змінюється з "Завантаження" на список звітів. Дані відображаються у таблиці.

2. Помилка мережі 
Дія: Зупинити бекенд сервер та оновити сторінку.
Очікуваний результат: Інтерфейс відображає повідомлення: Помилка: Failed to fetch . Система залишається стабільною.

3. Таймаут запиту 
Дія: Встановити штучну затримку на бекенді понад 10 секунд.
Очікуваний результат: Запит скасовується, відображається  Помилка.

4. Помилка 404 (Not Found)
Дія: Спробувати видалити звіт, який вже був видалений іншим користувачем.
Очікуваний результат: Виводиться повідомлення про помилку з відповідним HTTP-статусом.

5. Перевірка CORS
Дія: Відкрити консоль браузера (F12) під час завантаження даних.
Очікуваний результат: Запити проходять успішно, CORS помилки відсутні (дозволено лише конкретний origin фронтенду).

Приклади запитів для перевірки 
GET /api/v1/reports — отримання всіх звітів.
POST /api/v1/reports — створення нового звіту.
PUT /api/v1/reports/:id — оновлення існуючого звіту.
DELETE /api/v1/reports/:id — видалення звіту.
